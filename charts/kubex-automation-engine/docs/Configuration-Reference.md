# Configuration Reference

This document maps the current Helm chart values to the resources created by the chart and to the CRD-based runtime model.

## How to Use This Reference

- Use [Getting Started](./Getting-Started.md) for the install sequence
- Use this document to understand which values are required and which CRs Helm creates
- Use [Policy Configuration](./Policy-Configuration.md) for detailed strategy and scope examples
- Use [Cluster Automation Strategies](./Cluster-Automation-Strategies.md) and [Cluster Proactive Policies](./Cluster-Proactive-Policies.md) for cluster-scoped CRD field references

## Resources Created by This Chart

### Core Components

| Resource Type | Purpose |
| --- | --- |
| `Deployment` | Runs the controller manager and gateway sidecar |
| `Service` | Exposes metrics and webhook endpoints |
| `MutatingWebhookConfiguration` | Registers pod mutation webhook |
| `ValidatingWebhookConfiguration` | Registers validation webhook |
| `Secret` | Stores gateway credentials and TLS material when `createSecrets=true` |

### Generated Custom Resources

| Resource Type | When Created | Purpose |
| --- | --- | --- |
| `GlobalConfiguration` | `globalConfiguration.enabled=true` | Cluster-wide controller settings |
| `ClusterAutomationStrategy` | `policy.policies` defined | Cluster strategies generated by Helm |
| `ClusterProactivePolicy` | `scope` defined | Cluster-scoped proactive policies generated by Helm |

### Notes

- Helm-managed policy generation creates `ClusterAutomationStrategy` resources
- Helm-managed scope generation creates `ClusterProactivePolicy` resources
- `AutomationStrategy`, `ProactivePolicy`, `StaticPolicy`, `ClusterStaticPolicy`, `ClusterAutomationStrategy`, and `ClusterProactivePolicy` can all be managed manually outside Helm
- `StaticPolicy` and `ClusterStaticPolicy` are supported by the controller, but this chart does not currently generate them from values

## Required Values

| Key | Description |
| --- | --- |
| `kubex.url.host` | Kubex hostname, for example `example.kubex.ai` |
| `kubex.clusterName` | Cluster identifier presented to Kubex |
| `kubexCredentials.username` | Required when `createSecrets=true` |
| `kubexCredentials.epassword` | Required when `createSecrets=true` |
| `gateway.configSecretName` | Required when `createSecrets=false` |

## External Credential Secret

If you want to manage the Kubex credential secret outside Helm, disable Helm secret creation and point the gateway sidecar at a pre-created secret:

```yaml
createSecrets: false

gateway:
  configSecretName: kubex-gateway-config
```

Use `gateway.configSecretName` to reference the Secret name. This is the only chart value used to locate an externally managed gateway secret when `createSecrets=false`.

The Secret must:

- exist in the same namespace as the Helm release
- use the exact key names shown below
- keep the legacy `DENSIFY_BASE_URL` key because the current gateway image still expects it

Example:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: kubex-gateway-config
  namespace: kubex
type: Opaque
stringData:
  username: "<kubex-username>"
  epassword: "<kubex-encrypted-password>"
  url: "<instance>.kubex.ai"
  DENSIFY_BASE_URL: "https://<instance>.kubex.ai"
```

Keep `kubex.url.host` set in your values file so the release configuration still documents the target Kubex instance, even when the secret is managed outside Helm.

Note: `kubexCredentials.userSecretName` is currently not consumed by this chart. When `createSecrets=false`, set `gateway.configSecretName` instead.

## OpenShift Notes

- Installing this chart still requires cluster-scoped permissions because it creates `ClusterRole`, `ClusterRoleBinding`, and admission webhook resources
- If you deploy on OpenShift, enable `openshift.enabled=true` to apply restricted-friendly defaults for the cleanup job and optional supplemental groups; set `openshift.fsGroup` only when your storage class or SCC policy requires a fixed group
- The OpenShift compatibility switch is opt-in so standard Kubernetes installs keep the existing defaults

## Helm-Managed Policy Values

### `policy`

| Key | Description |
| --- | --- |
| `policy.defaultPolicy` | Default strategy name used when a scope omits `policy` |
| `policy.policies` | Map of strategy definitions generated as `ClusterAutomationStrategy` resources |
| `policy.policies.<name>.allowedPodOwners` | Comma-separated workload owner types copied into generated `ClusterProactivePolicy.scope.workloadTypes`. Supported values: `Deployment`, `StatefulSet`, `DaemonSet`, `CronJob`, `Rollout`, `Job`, `AnalysisRun`. `ReplicaSet` is not supported. |
| `policy.policies.<name>.enablement` | Resource change permissions and optional floor/ceiling bounds |
| `policy.policies.<name>.inPlaceResize` | In-place resize controls |
| `policy.policies.<name>.podEviction` | Eviction fallback controls |
| `policy.policies.<name>.safetyChecks.maxAnalysisAgeDays` | Recommendation freshness gate written into generated `ClusterProactivePolicy` |

These values apply to Helm-managed recommendation-driven automation only. If you need namespaced policies, fixed-resource policies, or independently managed cluster policies, create those CRs separately and reference either `AutomationStrategy` or `ClusterAutomationStrategy` as appropriate.

### `scope`

| Key | Description |
| --- | --- |
| `scope[].name` | Name of generated `ClusterProactivePolicy` |
| `scope[].policy` | Name of generated or externally managed `ClusterAutomationStrategy` to reference |
| `scope[].namespaces.operator` | `In` or `NotIn` |
| `scope[].namespaces.values` | Namespace selection values |
| `scope[].podLabels` | Label selector rules converted to `matchLabels` or `matchExpressions` |
| `scope[].weight` | Policy weight for precedence resolution |

For the generated resource fields behind these values, see [Cluster Automation Strategies](./Cluster-Automation-Strategies.md) and [Cluster Proactive Policies](./Cluster-Proactive-Policies.md).

| `deployment.controllerEnv.apiRequestTimeout` | `GlobalConfiguration.spec.kubexAPIRequestTimeout` |
